## 2. 各項目の検算式（第一原理）

```
lev_mod_065 = clip(lev_raw + clip(-k*lt*0.5, ±0.5), 0, 1),  k = vz>0.65?0.8 : vz<-0.65?0.1 : 0.5
            = clip(0.6108 + clip(-0.8*0.8549*0.5)) = clip(0.6108 - 0.3420) = 0.2688
W1          = OUT（初期OUT ∧ lev_mod 0.269 ≤ exit 0.3）
mom63_q     = 3（mom63 0.1456 > Q75 0.0910）
boost_mult  = BOOST_P09[3] = 1.00
L_t         = OUT → 0
bond_on     = (bond_mom252 -0.0043 > 0)? → 0
w_g         = (1/σg)/((1/σg)+(1/σb)), σg=0.2853 σb=0.0987 → 0.25695, clamp[0.25,0.75] → 0.257
holdings    = gold1x=w_g=0.257, bond1x= bondOff→0, cash=1-0.257-0=0.743
受渡         = lev_raw=Dyn2x3x new_leverage 0.6108, vz=Dyn2x3x vix_z 1.46
```

## 3. 当日シグナルの解釈（参考）

**P09 = OUT（最も防御的）**: ボラ高（vz=1.46）→ k_hi=0.8 で lev_mod_065=0.27 ≤ 0.3 → 株式OUT。債券モメンタムも僅かにマイナス（−0.43%）→ Bond OFF。
→ 防御スリーブ = **Gold 25.7% + 現金 74.3%**（Goldは逆ボラ配分、Bond分は現金待避）。これは設計どおり（現金過多はバグでない）。通知のフォワードリターンは OUT_bondOFF ビン = +1.3%/年。

## 4. 次アクション（運用乖離の統計評価へ）

- 数日〜2週間（5〜10営業日）分の P09Log が溜まったら本スクリプトを**全行で再実行**し、(a) 各項目の最大絶対差、(b) IN日が出た場合の TQQQ/CFD超過/ブースト点灯の検算、(c) GAS判定とPython日次再計算の不一致件数（目標0）を集計する。
